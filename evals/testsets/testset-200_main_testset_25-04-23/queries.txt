VRRP-based High Availability (HA) on Peplink Balance (210 and above) and MAX 700/HD devices allows two routers (master and slave) to provide seamless failover and hardware redundancy. The master router handles all connectivity and periodically sends VRRP advertisement packets (heartbeats) to the slave. How does IPv6 address assignment and routing differ from IPv4 in terms of address types, configuration methods, and routing protocol support, and what are the implications for devices connected to a Pepwave Balance One?

IPv6 address assignment differs from IPv4 by introducing new types of addresses and new methods for automatic configuration. IPv6 defines global unicast addresses (publicly routable), link-local addresses (starting with 'fe80', only for local communication), and unique local addresses (private but not internet-routable). Devices connected to a Pepwave Balance One receive both a link-local address (accessible only by devices on the same LAN port) and a global IPv6 address (accessible by any device on the LAN or internet), whereas IPv4 typically uses private addresses behind NAT. IPv6 addresses can be assigned manually, via DHCPv6, or automatically using SLAAC, which leverages router advertisements and the EUI-64 method to generate addresses. IPv6 eliminates the need for NAT for internal/external connectivity, although NAT64/NAT46 exist for interoperability. Routing in IPv6 supports both static and dynamic protocols, such as OSPFv3, with routing tables for IPv6 being independent from IPv4. OSPFv3 is configured differently, using interface assignments instead of network statements, and address families allow simultaneous support for IPv4 and IPv6. IPv6 also replaces ARP with Neighbor Discovery Protocol (NDP) for MAC address resolution and router discovery. The implications for Pepwave Balance One users are that devices will automatically receive both link-local and global IPv6 addresses, can participate in IPv6 routing without NAT, and benefit from improved auto-configuration and routing protocol support over IPv4.

How does Peplink's SpeedFusion technology manage and optimize multiple WAN connections for both reliability and performance, and what are the key configuration and architectural considerations for deploying it in diverse environments?

Peplink's SpeedFusion technology is designed to bond multiple WAN connections—including cellular, fiber, DSL, satellite, and Wi-Fi—into a single robust connection that enhances both reliability and bandwidth. At its core, SpeedFusion establishes concurrent VPN tunnels between at least two Peplink devices (or a Peplink device and FusionHub/SpeedFusion Cloud), distributing traffic at the packet level to maximize available bandwidth and provide seamless failover. This means if one WAN fails, traffic is instantly rerouted through healthy connections, ensuring uninterrupted sessions, which is crucial for real-time applications like VoIP and video conferencing.

Key features include:
- Bandwidth Bonding: Aggregates throughput from multiple links into a single high-speed connection, which is especially beneficial when individual WAN links are insufficient for application needs.
- Hot Failover: Instantly switches sessions to another available WAN during a failure, preventing downtime.
- WAN Smoothing: Sends duplicate packets across multiple WANs to minimize packet loss and jitter, enhancing quality for sensitive applications.
- Forward Error Correction (FEC): Adds parity bits to reduce packet loss without excessive bandwidth overhead.
- Outbound Policies and Sub-Tunnels: Administrators can define traffic distribution rules and create sub-tunnels with specific settings (e.g., lowest latency, weighted round robin) to optimize performance for different application types.
- Dynamic Weighted Bonding: Adjusts traffic allocation in real-time based on link quality, allowing for optimal use of diverse WAN types (e.g., Starlink and cellular).

Architectural and configuration considerations include:
- At least two Peplink endpoints are required for bonding (e.g., branch router and HQ router, or router and FusionHub/SpeedFusion Cloud). One endpoint should have a public IP address.
- SpeedFusion supports various topologies (point-to-point, star, mesh) and can be integrated with existing IPsec VPNs.
- Device selection should match bandwidth needs; higher-end models support higher aggregate throughput.
- The InControl2 platform provides centralized management, zero-touch provisioning, reporting, and remote troubleshooting.
- Outbound policies can be tuned to steer traffic based on application needs and link conditions.
- Inclusion of unstable or highly variable WAN links may negatively impact overall performance, so careful configuration and ongoing monitoring (e.g., using WAN Analyzer or SpeedFusion graphs) is essential.
- SpeedFusion supports seamless integration with SD-WAN and can be deployed incrementally without disrupting existing networks.

In summary, SpeedFusion’s flexible architecture and advanced traffic management features enable organizations to achieve unbreakable connectivity and optimized performance across a range of deployment scenarios, from remote sites and mobile environments to branch offices and data centers.

Configuration Sync can be enabled to automatically synchronize settings from the master to the slave every five minutes, or the slave can be configured manually. For resilience, additional Ethernet switches are recommended to avoid a single point of failure. In NAT mode, the VIP must be set as the default gateway for all LAN hosts, and both routers should be connected to the internet in NAT mode for proper operation.

In a Peplink Balance HA setup using VRRP, what are the necessary steps and configuration requirements to ensure seamless failover, and what common pitfalls should technicians avoid, especially regarding network topology and device synchronization?

To ensure seamless failover in a Peplink Balance High Availability (HA) setup using VRRP, both routers must be configured in a master-slave (active-standby) architecture, where one device is designated as the master and the other as the slave. Both devices must be on the same subnet, share the same HA group identifier, and use the same operational mode (either NAT or Drop-in). The Virtual IP (VIP) must be set as the default gateway for all LAN hosts. Configuration Sync can be enabled for the slave to automatically inherit settings from the master (with sync intervals such as every five minutes), or the slave can be configured manually. During normal operation, the master sends periodic VRRP advertisement packets to a multicast address; if these are not received by the slave within 10-15 seconds (the typical failover window), the slave takes over, re-establishes WAN connections, and binds the VIP for LAN connectivity. Upon recovery, the master resumes its role. Use additional Ethernet switches to avoid single points of failure in the network topology. A common pitfall is failing to synchronize configurations, which can result in mismatched settings and failed failover. Another is not ensuring both devices are on the same subnet or not using the same HA group identifier. Technicians should also avoid using peer-to-peer HA topologies, as Peplink's HA implementation only supports master-slave operation. In some cases, issues can arise if the VRRP MAC address is not correctly set or if both routers attempt to assume the master role, especially after connectivity disruptions; careful diagnostics and physical connection checks are recommended to resolve such issues.

To mitigate this, it is a common and effective best practice to switch WAN DNS servers and health check targets to reliable public DNS providers such as Google (8.8.8.8 and 8.8.4.4), which improves the reliability of health checks and reduces unnecessary disconnects.

For further reliability, it is advisable to configure both a primary (typically the ISP DNS for low latency) and a backup public DNS server, ensuring redundancy in case one provider fails. Additionally, on some Peplink models, health check methods can be changed from DNS lookups to pinging a website or the WAN's default gateway, which may offer more consistent results and help avoid false negatives, especially if repeated DNS queries risk being blocked by the ISP or if the DNS server itself is unreliable.

Disabling health checks entirely can be a temporary workaround for persistent false disconnects, but it may result in unnoticed outages. Therefore, maintaining health checks with reliable targets is preferable for critical environments. When using multiple WANs, proper configuration of DNS and health check targets, alongside accurate WAN bandwidth settings and priority algorithms, further enhances stability and performance. Lastly, Peplink devices do not serve as DNS servers themselves and only alert on total DNS failures, so using third-party DNS providers with failover capabilities is recommended for environments requiring high uptime.

How does SpeedFusion Cloud differ from FusionHub in terms of setup, flexibility, and application scenarios, and what are the key considerations for optimizing connectivity and application performance when using SpeedFusion Cloud on a Pepwave device?

SpeedFusion Cloud is designed for ease of use, enabling users to bond multiple internet connections for seamless connectivity with minimal setup and no need for additional hardware or technical expertise. Users can activate it through their Pepwave device's interface, often out-of-the-box or with a simple activation key, and benefit from features like bandwidth bonding, automatic failover, load balancing, WAN smoothing, and centralized management. It is ideal for mobile/remote workforces, branch offices, critical operations, and environments where quick deployment and reliable internet are essential. The service supports multiple tunnels, prioritization of applications, and SSID-based traffic management, with built-in global cloud node selection for optimized latency. However, it is limited to up to 50 Mbps per device and uses shared IP addresses, with no inbound services.

FusionHub, on the other hand, is a virtual appliance that can be self-hosted on preferred platforms (cloud or on-premises), offering greater flexibility for advanced users and businesses. It allows for custom deployment locations to reduce latency, provides dedicated static IPs, supports granular control over latency and packet loss, and integrates with other resources or infrastructure like AWS Transit Gateway Connect. This makes it more suitable for larger business needs, advanced routing, and scenarios where inbound access or custom networking is required.

Key considerations for optimizing SpeedFusion Cloud performance include:
- Using devices with firmware 8.1 or higher and at least two active internet connections for effective bonding and failover.
- Prioritizing traffic or applications (like Zoom, Teams, Netflix) via outbound policy management, SSID mapping, or device-based routing.
- Grouping WAN connections with similar latency to avoid performance degradation when bonding (especially when mixing technologies like LTE and satellite).
- Disabling WAN smoothing if bandwidth conservation is desired, as this duplicates packets and can increase data usage.
- Monitoring WAN quality reports and using external antennas to improve cellular signal strength, if needed.
- Recognizing that SpeedFusion Cloud is best for outbound (internet-bound) traffic and may not support inbound services or static IPs.
- Considering FusionHub or physical appliances for scenarios requiring advanced routing, custom deployment, or inbound access.

Overall, SpeedFusion Cloud simplifies SD-WAN and bonded connectivity for most users, while FusionHub provides advanced options for those with complex business or technical requirements.

How can a technician configure a Peplink Balance 20/30 series router to ensure specific machines or types of traffic are routed only through designated WAN connections, and what are the effects of the default settings versus custom rules?

On a Peplink Balance 20/30 series router, to ensure that specific machines or types of traffic are routed exclusively through designated WAN connections, the technician should adjust the outbound policy settings. By default, the outbound policy is set to 'Auto', which uses a weighted balance algorithm based on the downstream bandwidth of each WAN interface. This means traffic is distributed according to the available bandwidth, and there is no guarantee that a specific machine or type of traffic will always use the same WAN connection, potentially causing session integrity issues for protocols like FTP.

To override this, the technician must change the outbound policy to 'Custom Rules' and retain the default 'HTTPS Persistence' and 'Default' rules. Then, new rules should be added specifying the source (such as a particular LAN client or traffic type) and the WAN connection that should be used. This ensures that traffic from specified devices or applications is routed only through the intended WAN, avoiding round-robin distribution and maintaining session integrity. After creating these rules, it is essential to save and apply the changes. This setup allows granular traffic routing, such as ensuring all FTP traffic from a specific server always exits through WAN1, and that return traffic is correctly routed back on the same link.
